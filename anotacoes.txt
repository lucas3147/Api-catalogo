API - Application Programming Interface

Conjunto de rotinas e padrões estabelecidos e documentados por uma aplicação para que outras aplicações consigam utilizar suas funcionalidades sem precisar conhecer detalhes da sua implementação


1 - Permitem fazer a integração de sistemas

2 - Faciltam o intercâmbio de informações 

3 - Oferecem maior Segurança nos dados

4 - Permite fazer o Controle de Acesso

5 - São fáceis de implementar e usar

WEB SERVICES

	São um conjunto de métodos inocados por outras aplicações utilizando tecnologias WQeb sendo uma solução utilizada na integração de sistemas e na comunicação entre aplicações distintas

Características

	1 - Estão apoiados no XML e no protocolo SOAP

	2 - Permitem integração de sistemas

	3 - Permitem a reutilização de código
	
	4 - Podem ser desenvolvidas em tempo reduzido

	5 - Apresentam maior segurança

EVOLUÇÃO DOS WEB SERVICES PARA WEB API

WEB API

	É uma API na internet, com um conjunto de serviços expostos via web ara realizar a integração da aplicação com diversos tipos de clientes (web, mobile, desktop, etc.)

1 - Java, .NET, Ruby On Rayls, JavaScript, Node.js, etc.
2 - Resposta em um formato específico como CML, JSON, etc.
3 - Pode ser acessadas usando o protocolo HTTP
4 - Tipos de resposta : arquivo texto, imagem, som, operação, banco de dados, objetos, etc.

				AULA 10: NÍVEIS DE MATURIDADE DE RICHARDSON

É um modelo sugerido em 2008 por Leonard Richardson que classifica as APIs com base em sua aderência e conformidade com cada um dos quatro níveis do modelo :

Nível 0:

	POX - Plain Old XML

Nível 1: 

	Recursos

Nível 2: 

	Verbos HTTP

Nível 3 (conhecido como a glória do Rest):

	HATEOAS

A maturidade de um servoço é baseada em três fatores neste modelo: Recursos / URI, verbos HTTP e HATEOAS (Hipermídia)

// Na teoria, apenas as APIs que atendem o nível 3 (HATEOAS) podem ser consideradas APIs Rest

// Na prática, podemos classificar grande parte das APIs como estando no nível 2
